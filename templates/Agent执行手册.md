# Agent执行手册

本手册规定 agent 每次做任务时的操作纪律。用户负责提出任务和纠偏；agent 负责执行、归类、记录、提炼和回填。

## 任务分级 · 流程重量

流畅等级（A/B/C/D，见 `README.md`）决定**读多少**；任务分级决定**走多重的流程**，两者正交。开工前先定级，避免小事走全套流程。

| 级别 | 何时用 | 规则 |
|---|---|---|
| P0 · 只路由 | 用户只问"该用哪条流程 / 下一步做什么" | 直接按 `快速入口.md` 判断，不写记录、不建台账 |
| P1 · 普通任务 | 查询、解释、审一段内容、一次性改小字段 | 装最相关流程直接做；不需要的台账一律不碰 |
| P2 · 轻量变更 | 改入口措辞、已有流程小补丁、台账同步、候选流程小修 | 先改 + 回填；只在改动面变大时才升级处理 |
| P3 · 正式变更 | 新工作流入宪、改启动顺序 / 归类默认 / 清理边界、大流程改造 | 才走完整流程：评分 → 候选验证 → 巡查确认 → 回填多个入口 |

**默认降级原则**：能 P1 不 P2，能 P2 不 P3。完整流程只用于会改变系统规则、入口或长期制度的事。

**P2 累积触发线**：同一会话内的轻量改动累计触及 **3 个及以上**入口 / 工作流 / 台账，或把临时资料接进 `AGENTS.md` / 正式工作流长期引用 → 必须升 P3 处理（巡查确认后再固化）。这条防的是"每次都说轻量、攒到一起其实动了半个系统"。

## 每次任务流程

1. 读取 `README.md`，判断是否命中已知工作流和流畅等级。
2. A 级完全稳定流程只读执行卡片和必要公共文档，不默认读取本手册后续长规则。
3. B/C 级、异常、新任务、路由不清或维护任务，再查看 `快速入口.md`、`系统索引/工作流总览.md` 和 `巡查/巡查日志/` 最新一份。
4. 命中已有 `workflow.md` 时，按流畅等级决定是否给执行前的原工作流评分。
5. 上表暂无对应项时，创建或更新 `工作流/{任务名称}/candidates.md`。
6. 执行任务，每一步记录输入、动作、成功读回、失败分支和风险点。
7. 用户纠正时，按 `系统索引/用户纠正处理规则.md` 归类。
8. 产生文件或内容时，按 `系统索引/产出归类与命名规范.md` 放置。
9. 涉及网页或外部系统时，补充 `DOM地图/{系统名}.md`。
10. 结束时写 `执行记录/原始记录/{任务名}_{YYYYMMDD_HHMM}.md`。
11. 如果出现可复用步骤，写入候选流程；巡查验证后再进入主工作流、快速入口或巡查日志。

## agent 接手 / 可交接性检查

目标不是多派 agent，而是验证"下个 agent 只按文档能不能接手"。B/C/D 级、新流程、批量任务、外部系统任务、A 级异常或新 agent 卡住时触发；A 级普通复用不默认触发。

接手前先回报 5 件事：

1. 命中的流程和流畅等级。
2. 已读资料：`workflow.md` / `candidates.md` / DOM 地图 / 资料库 / 最新巡查相关段落。
3. 必要输入和权限是否齐。
4. 风险停点：提交、删除、覆盖、写线上数据、付款等动作前停在哪里。
5. 成功读回：完成后看哪个字段、日志、文件、数量或页面状态。

验证方式：

- 新流程或批量任务先跑 1 个最小样本；能成功读回再扩大范围。
- 文档写好后，可派 subagent 只按文档试跑 1 个最小样本；subagent 卡住 = 文档没写清。
- 卡住时不先加制度，先补最小缺口：`candidates.md` / `workflow.md` / `DOM地图/{系统名}.md` / 资料库对应页。
- 执行记录里写清"本次是否能按文档接手、卡在哪里、应补哪个文件"。

## 评分触发

A/B/C/D 流畅等级和读取范围以 `README.md` 为准。本手册只规定评分何时发生：

- A 级完全稳定：默认不评分；异常、页面变化、字段变化、新 agent 卡住时评分。
- B 级基本稳定：卡顿、用户纠正或候选规则不清时评分。
- C 级已做过但未稳定：执行前建议评分，用于判断是否小样本执行。
- D 级未命中：不做原工作流评分，先建立候选流程。

## 原工作流评分

评分对象是本次执行前已经存在的 `workflow.md`，不能用执行后补充的版本倒算。评分按流畅等级触发，不要求每次复用都评分。

| 维度 | 评分 0-5 | 低分信号 |
|---|---:|---|
| 触发和适用场景 |  | 不清楚什么任务该用这个流程 |
| 必要输入 |  | 缺少数据来源、字段、账号或文件要求 |
| 执行步骤 |  | 步骤太泛、顺序不清、无法照做 |
| 成功读回 |  | 只写动作，没有写怎么看是否成功 |
| 失败分支 |  | 页面变了、字段缺失、权限失败时不知道怎么办 |
| 产出归类 |  | 不清楚正式产出、验证证据、临时文件放哪里 |
| 风险控制 |  | 提交、删除、写入线上数据前没有确认规则 |

总分判断：

| 总分 | 判断 | 执行动作 |
|---:|---|---|
| 0-15 | 原工作流不可直接执行 | 先补流程或问用户 |
| 16-24 | 小样本执行 | 边执行边补成功读回和失败分支 |
| 25-31 | 可按流程执行 | 执行后补低分项 |
| 32-35 | 较稳定 | 进入固化或工具化候选 |

## 新任务流程

1. 明确当前要交付的结果。
2. 建立 `工作流/{任务名称}/candidates.md`，使用 `工作流/_候选流程规则.md`。
3. 先完成最小可验证步骤。
4. 把成功读回、失败分支、风险点和工具化线索写进候选流程。
5. 如果探索了网站或外部系统，更新 `DOM地图/{系统名}.md`。
6. 写执行日志，并在 `快速入口.md` 记录候选路由（等巡查确认后升级为正式行）。

## 工作流训练环

训练目标是让流程越跑越短、越准、越能交接，不是训练"专注力分数"。B/C/D 级、新流程、近期卡点流程和准备升 A 的流程触发；A 级稳定流程不默认增加负担。

1. **开工前 30 秒**：用 5 行复述入口、必要输入、标准步骤、成功读回、风险停点；同时选 1 个本轮训练点（如成功读回、失败分支、风险停点、产出归类）。
2. **执行中只练 1 个弱项**：不要每轮都加新指标。低分项来自原工作流评分、用户纠正或上次巡查。
3. **收尾 3 行**：本轮漏了什么；下次触发句是什么；是否值得写入 `candidates.md` / `workflow.md`。
4. **巡查压缩**：2-3 次真实任务验证过的训练点进主流程；未验证的只留在巡查待验证项。

不要引入通用脑力训练、番茄钟打卡、动机口号或长篇学习笔记；凡是不能变成触发、动作、读回、停点的内容，都不进入工作流。

## 专项训练模式

专项训练只用于已经有轮廓、但执行慢、脆弱、反复卡住的流程。目标是把主路径打磨到"基本可运行模块"，不是把所有边界一次练完。

触发条件：

- 同一流程反复卡在入口、选择器、文件位置、登录状态、成功读回或提交边界。
- B/C 级流程已经能描述主路径，但新 agent 或 subagent 按文档跑不顺。
- 批量、网页、外部系统或连接器流程需要先验证 1 个样本。

默认刹车：

- 每次专项训练最多 3 轮 subagent 验证。
- 每轮只用 1 个真实或已完成样本；主路径跑通后最多再加 1 个交叉样本。
- 主 agent 调查单个专项训练最多 30 分钟，超过就给状态报告。
- 主路径达到"基本可运行模块"就停；罕见分支写入未来执行日志，不在本轮穷举。

1 分钟规则：

- 如果 routine 步骤 fresh subagent 1 分钟内跑不动，先当作流程缺口，不先怪 subagent。
- 缺口通常是入口不清、旧状态没重置、DOM/文件地图不够、权限/账号没写、成功读回太泛、失败分支缺失、提交边界不清。
- 外部等待、登录审批、系统慢响应不算 routine 卡点，但要记录为外部等待或权限条件。

subagent 验证提示词：

```text
Use the workflow at <absolute workflow path> to process/check this sample: <sample>.

Boundary:
- Do not submit / Do not save / Do not send unless explicitly stated.
- Stop and report if any step takes more than 1 minute.
- For each step, report: step id/name, elapsed time, what you tried, what readback you saw, and the first unclear or missing instruction.
- Do not optimize the workflow yourself; follow it as written and report where it is insufficient.
```

卡点类型：

- 入口或登录 profile 错。
- 旧筛选、旧页面、旧文件状态没有重置。
- selector、按钮、字段、tab、行定位不清。
- 结果打开了错误对象。
- 成功读回缺失或太宽。
- 字段显示了值，但没有真正选中。
- 输出受编码、控制台或临时文件位置影响。
- "检查"没有写清哪个信号证明成功。
- live side-effect 边界不清。
- subagent 必须自行发明 workaround。

收尾输出：

- 稳定主路径步骤。
- 仍然存在的卡点。
- 延后到未来执行日志的冷门分支。
- 是否值得以后再开一轮专项训练。

## 收尾检查

- 正式产出、验证证据、临时文件是否已归类。
- 多版本文件按 `系统索引/产出归类与命名规范.md` 处理，确认稿进入 `output/正式产出/`，旧版本移到 `_待删除/` 或归档。
- 用户纠正是否已写入执行日志或候选规则。
- DOM 地图、资料库、快速入口是否有需要更新的内容。
- 是否记录了原工作流评分和低分原因。
- 是否出现可工具化动作。
- 是否还有需要用户确认的风险动作。
- 若该工作流连续完整执行不卡点已达第 5 次，**当场在对话框问用户**要不要升极速模式（用户拍板，不替用户决定）；**用户没回答就当没看到，下次执行完再问，只有明确选了要/不要才定**。极速分档与升降级见 `完整工作手册.md`「极速模式」。

## 网页 / 批量任务规则

### 入口确认协议

网页任务开工前先回报 3 件事：
1. 读了哪些文档（DOM地图、流程文件）。
2. 是否在正确页面（读回页面标题或关键字段）。
3. 待处理数量 + 第一条的字段预览。

入口确认最多 2 分钟，超时派 subagent 复核，不自己瞎摸。

### 1 样本先跑通 + subagent 试跑

1. 手动跑通 1 条。
2. 把流程写成文档。
3. 派 subagent 只按文档跑 1 条。
4. subagent 卡住 = 文档没写清 → 补文档 → 才批量。
5. 新流程默认停在提交前，等用户确认。

### DOM 铁律

- 不用固定屏幕坐标。
- 定位优先级：稳定选择器 > 可见文本 > 按钮文本 > 元素内相对位置 > 最后才人工坐标。
- 每次点完读回字段值，不信"点了就成功"。

### 三层拆分

| 层 | 内容 | 位置 |
|---|---|---|
| DOM 地图 | 元素在哪 / 怎么稳定找 | `DOM地图/{系统名}.md` |
| 公共流程 | 同系统多任务共用的入口路径 | `工作流/{名称}/workflow.md` 公共段 |
| 分支流程 | 分支点后的差异步骤 | `工作流/{名称}/workflow.md` 分支段 |

同系统新任务先复用公共流程 + DOM 地图，只学分支点之后的差异。

### 日志颗粒度

- 不是每次点击都写，按关键对象记录。
- 每条记录/文件 1 行，批量每 3–5 条小结，失败/变更单独记。
- 可逆·极速工作流：不卡点可不写执行日志；不可逆或出异常的，至少写一行回执。
