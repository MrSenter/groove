# 巡查执行流程

目标：把原始执行记录压缩成更短的入口、等级、执行卡片、候选规则，沉淀成一份**巡查日志**，让下次执行少读内容。

巡查日志按日期滚动：每次巡查在 `巡查/巡查日志/` 新增一份 `巡查日志_{YYYYMMDD_HHMM}.md`。**最新一份就是当前结论**，旧版本全部保留——执行日志被清理后，这些巡查日志就是改动历史和经验精华，也是回看"我们改了什么"的唯一记录。

## 什么时候执行

- 每周固定巡查。
- 同一流程连续出现卡顿。
- 用户纠正了可复用规则。
- 新 agent 按文档执行时卡住。
- 页面、字段、工具或输出规则发生变化。
- 多轮 v1/v2/v3 版本已经产生确认稿，需要处理旧版本。

## 输入

| 输入 | 读取方式 |
|---|---|
| `巡查/滚动未决.md` | **最先读**，这是跨巡查的未决项常驻台账（待验证 / 待确认 / 观察中）；逐条写使用反馈、更新状态 |
| `巡查/巡查日志/` 最新一份 | 再读，确认上次结论和本轮改动史 |
| `执行记录/原始记录/` | 只读本周期新增记录，必要时追溯旧记录 |
| `工作流/{名称}/workflow.md` | 判断主流程是否需要更新 |
| `工作流/{名称}/candidates.md` | 判断候选规则是否已验证（按需存在） |
| `DOM地图/` | 只在页面路径、字段、按钮变化时读取 |
| `资料库/` | 只在固定规则、账号权限、背景变化时读取 |
| `临时内容/` | 只检查待处理版本稿、验证文件和中间文件是否还需要保留 |

## 执行步骤

1. 确定巡查周期和涉及工作流。
2. **先读 `巡查/滚动未决.md`**：逐条回看未决项，写入真实使用反馈（是否提速 / 更顺 / 新卡点），更新状态并初步判断 固化 / 继续观察 / 回滚；拖过 3 轮还没动的必须当轮给交代。再读上一份巡查日志确认改动史。
3. 读取本周期新增原始记录，按工作流聚合。
4. 提取重复卡点、用户纠正、成功读回、失败分支、工具化线索。
5. 统计执行记录：本周期记录数、高频错误（≥2 次）、高频卡点、训练点验证情况、哪些工作流可升级为稳定（A 级）、哪些仍需小样本验证。**维护每个工作流"连续完整执行不卡点次数"：达到 5 次而 agent 当时漏问的，巡查补提一次升极速；本周期出过异常的清零并记降级。**
6. 新建本次 `巡查/巡查日志/巡查日志_{YYYYMMDD_HHMM}.md`（用 `_巡查日志模板.md`），填：发现的问题、已固化的工作流、未定下来的内容、本次改动方案、执行日志统计。**这份日志本身就是方案。**
7. **在对话框里把"本次改动方案"逐条念给用户，确认哪些改动落地、哪些内容保留或丢弃。** 确认在对话框直接发生，不靠额外计划文件走流程。
8. 按用户确认结果落地：已确认改动写入 `workflow.md`、执行卡片、DOM 地图或资料库；更新 `快速入口.md`、`系统索引/工作流总览.md` 的流畅等级和成熟度；把确认结果回填进本份巡查日志的"本次执行的改动"表。
9. **更新 `巡查/滚动未决.md`**：本轮新增的未决项（待验证 / 待确认 / 观察中）追加进常驻台账；本轮关掉的标"已解决"+落地位置。本份巡查日志的"未定下来的内容"只记本轮快照，权威清单以滚动未决.md 为准。
10. 标记已吸收原始记录；被吸收后不再进入日常默认读取路径（精华已进巡查日志）。
11. 检查 `临时内容/` 中的 v1/v2/v3、draft、review、修改稿：确认稿已进入 `output/` 后，旧版本默认移到 `_待删除/`；需要追溯才移入 `归档/`。

## 输出

| 输出 | 作用 |
|---|---|
| `巡查/滚动未决.md` | 跨巡查的未决项权威清单（已更新状态）；下轮巡查的第一读 |
| `巡查/巡查日志/巡查日志_{时间}.md` | 本次巡查的结论 + 改动方案 + 执行统计；最新一份即当前记忆入口，旧份留作改动史 |
| `快速入口.md` / `系统索引/工作流总览.md` | 更新默认入口和流畅等级 |
| `workflow.md` / `candidates.md`（按需） | 固化或保留待验证规则 |

## 待验证优化 + 使用反馈环

每条优化落地先在巡查日志里标"待验证"，记录：
- 优化项：做了什么改动
- 预期效果：期望达到什么结果
- 验证场景：下次什么任务能验证

下次巡查必须回看 `巡查/滚动未决.md` 的"待验证"项，写入真实使用反馈后，再决定：
- 固化（写入主流程）
- 继续观察（保留待验证标记）
- 回滚（注明原因）

## 判断标准

| 情况 | 动作 |
|---|---|
| 规则已被真实任务验证 | 写入主流程或执行卡片 |
| 规则有用但未验证 | 标"待验证"留在本份巡查日志，记录预期效果和验证场景 |
| 只是单次失败过程 | 留在巡查日志的"发现的问题"，不进入主流程 |
| A 级流程无异常 | 不增加默认读取内容 |
| A 级流程连续 5 次不卡点 | 提示升极速（执行完第 5 次时由 agent 问用户；巡查补漏。可逆/不可逆分档见 `完整工作手册.md`） |
| 极速流程出现异常 | 退回普通模式 + 恢复完整日志 + 连续次数清零，并告知用户 |
| B/C 级流程仍卡顿 | 在巡查日志保留提醒，必要时降级 |
| 训练点被 2-3 次真实任务验证 | 压缩进 `workflow.md`；未验证继续留待验证项 |
| 训练点只是口号或无法读回 | 不进入主流程，必要时删除 |
| 专项训练主路径已跑通 | 固化主路径；冷门分支写入未来执行日志，不继续穷举 |
| 专项训练 3 轮仍未跑通 | 输出状态报告，标明卡点和是否值得下次继续 |
| 确认稿已经产生 | 旧版本移到 `_待删除/` 清理或移入 `归档/`，不进入 output |

## 收尾检查

- 是否更新了流畅等级（在 `快速入口.md` 和 `系统索引/工作流总览.md`）。
- 是否说明哪些旧原始记录不再默认读取。
- 是否更新了 `巡查/滚动未决.md`：新未决项已追加、本轮解决的已标"已解决"+落地位置、拖 3 轮的已给交代。
- 是否把已验证训练点压缩进主流程，未验证训练点继续留待验证项。
- 是否把已确认规则写入更短的入口或主流程。
- 是否避免把流水账复制进巡查日志（只留压缩后的结论和统计）。
- 是否处理了已确认稿对应的旧 v1/v2/v3 版本。
- 是否回看了 `巡查/滚动未决.md` 的待验证项并写入使用反馈。
- 是否填了执行日志统计（高频错误 / 可进稳定的工作流）。
