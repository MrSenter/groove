# 候选流程规则

候选流程用于记录新任务的初步流程。目标是让下个 agent 少摸索，成熟后能直接升级为 `workflow.md` 或脚本。

## 何时创建

- `README.md` 快速入口暂无对应项，且任务出现可重复步骤。
- 用户纠正了某个步骤、判断或输出格式，且未来可能复用。
- 任务涉及网站、外部系统、批量处理、文件生成、数据核对或固定格式产出。

## 保存位置

新建或更新：

```text
工作流/{任务名称}/candidates.md
```

如果同一任务已经有目录，追加本次新发现；如果没有目录，先创建候选目录。

## 记录精度

候选流程要写到接近脚本化的精度：

| 内容 | 要写清楚什么 |
|---|---|
| 触发说法 | 用户可能怎么说这个任务 |
| 必要输入 | 文件、字段、编号、账号、页面、日期、数量 |
| 稳定入口 | 从哪里进入页面、目录、表格或系统 |
| 参数 | 哪些值每次变化，哪些值固定 |
| 标准动作 | 每一步的动作和顺序 |
| 成功读回 | 页面字段、日志、文件、数量、状态如何确认成功 |
| 失败分支 | 找不到入口、字段变化、权限不足、数量不一致时怎么处理 |
| 风险动作 | 哪些动作需要用户确认 |
| 地图更新 | 本次探索补充到了哪个 `DOM地图/{系统名}.md` |
| 脚本线索 | 可自动化步骤、脚本参数、检查断言 |
| 接手验证 | 下个 agent 接手前读什么、先跑哪个最小样本、卡在哪里 |
| 训练点 | 本轮只练哪个弱项，收尾发现了什么，下次触发句是什么 |

## 升级条件

- 同类任务复用 2 次：保留候选流程并补充失败分支。
- 同类任务复用 3 次：整理为 `workflow.md` 草案。
- 有稳定脚本或自动检查：成熟度可升到 L4，并加入 `README.md` 快速入口。
